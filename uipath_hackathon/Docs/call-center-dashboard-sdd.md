## 1. App Overview

The supervisor/analyst-facing dashboard: 8 headline KPIs, 6 analytics drill-down areas, follow-up lifecycle management, and an AI Insights chat. Ports all 17 routes and the full component tree  §7, swapping the data layer from REST-over-Postgres to direct Data Fabric reads (+ one Agent call for chat).

### In Scope

All 17 routes from the existing `App.tsx`: Dashboard, Call Log, Call Detail, Agents, Agent Detail, Sentiment, Escalations, Compliance, Resolution, Intents, Trigger Words, Friction, Follow-ups, Follow-ups Overview, AI Insights, Live Call, Marketing, Login.

### Out of Scope

- Authentication implementation detail (Coded Apps inherit UiPath Identity — the existing `LoginPage` becomes a thin wrapper, not a custom auth flow)
- Any AI computation (delegated entirely to `call-center-ai-agent`)

## 6. API Integration

### UiPath API Workflow Calls

Not used — this app calls Data Fabric directly via the `@uipath/uipath-typescript` SDK and invokes `call-center-ai-agent` directly for the `ask` tool. No intermediate API Workflow layer is needed since Data Fabric's native query API covers the existing REST endpoint shapes (filters, pagination, aggregation).

| As-built endpoint | Coded App equivalent |
|---|---|
| `GET /api/calls`, `/api/calls/{id}` | Data Fabric `CallRecord` query / get-by-id |
| `GET /api/dashboard/summary` | Data Fabric aggregation query |
| `GET /api/agents`, `/api/agents/{name}` | Data Fabric `CallRecord` aggregation |
| `GET /api/analytics/*` (20+ endpoints) | Data Fabric aggregation queries, or delegated to `call-center-ai-agent`'s analytics tools where the existing logic lives in Python (friction scoring, marketing scoring) rather than pure SQL |
| `POST /api/ai/ask` | Direct call to `call-center-ai-agent.ask` |
| `GET /api/followups` | Data Fabric `CallFollowup` query (read) |
| `POST/PATCH/DELETE .../followups/*` | `call-center-followup-case` HITL forms (approve/reject/assign/track/complete) — see `call-center-followup-case-sdd.md` §5 |

## 7. User Flows

### Flow: Supervisor reviews and approves a follow-up

1. Supervisor opens `/followups/overview`, sees pending items (Data Fabric query, sorted overdue-first)
2. Clicks Approve → surfaces the `Supervisor Review`/`Assign Details` HITL forms owned by `call-center-followup-case` (the Coded App embeds that process's forms, it does not reimplement the approval logic — see `call-center-followup-case-sdd.md` §5)
3. Item moves through `in_progress` → `completed` as staff update status via the `Track Progress`/`Mark Completed` forms

### Flow: Supervisor asks an analytics question

1. Supervisor opens `/ai-insights`, picks a suggested question or types one
2. Coded App calls `call-center-ai-agent.ask(question)` directly
3. Markdown answer renders with `TypewriterText` animation; `AiUsageBadge` reflects the incremented counter

## 8. Error Handling

| Scenario | Handling |
|---|---|
| Data Fabric query failure | `EmptyState` / `LoadingSpinner` components show error state, ported as-is from existing `shared/` components |
| Agent (`ask`) call failure/timeout | Chat UI shows a retry affordance; does not crash the page |
| Stale filter state across navigation | `FilterContext` persists across route changes, same as existing |

## 11. Testing Strategy

### Canonical Test Cases

- Dashboard loads and renders all 8 KPI hero cards with correct RAG status against the 6 benchmarks
- Call Log filters (date range, agent, sentiment, search) return correctly filtered results
- Follow-up approve/reject/assign/complete actions correctly reflect HITL-driven status changes

### Unit Tests

Per-component tests for the 18 chart components (data-shape contract tests) and the benchmark-status utility (`getBenchmarkStatus`), ported from existing logic.

### E2E Tests

Full supervisor journey: load dashboard → drill into a call → approve a follow-up → ask an AI Insights question → confirm `AiUsageBadge` increments.
