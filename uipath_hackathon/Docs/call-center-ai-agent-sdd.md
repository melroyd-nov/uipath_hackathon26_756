## 1. Agent Overview

A single Coded Agent exposing 14 tools: 7 per-call AI extraction tasks, 6 analytics aggregation modules, and 1 grounded Q&A tool. This consolidates `ai_service.py` and `analytics_service.py` from the existing system into one agent rather than splitting into multiple agents, because all 14 tools share the same Data Fabric read access and the same LLM Gateway connection — there is no isolation boundary that would justify separate agents.

### In Scope

- 7-way per-call AI extraction (sentiment, intents, trigger words, compliance, follow-ups, resolution flags, summary)
- 6 analytics aggregation modules (sentiment, escalation, compliance, resolution, intent/marketing, friction/trigger-words)
- Grounded Q&A (`ask`) using a live Data Fabric snapshot as context, mirroring `_build_context()`
- AI usage logging (`calls_today`/`total_calls` counters)

### Out of Scope

- Pipeline sequencing (owned by `call-center-orchestration`)
- UI rendering (owned by `call-center-dashboard`)
- Direct telephony/transcript-file handling (the Flow passes already-read text; this agent does not touch the storage bucket)

## 2. Agent Framework

**Framework:** Simple Function (tool-calling Python functions), not LangGraph/LlamaIndex/OpenAI Agents — the existing `ai_service.py` has no multi-step planning or agentic loop; each tool is a single LLM call (extraction) or a single SQL-aggregation-then-LLM call (Q&A). A heavier framework would add orchestration overhead with no corresponding benefit here, since `call-center-orchestration` already owns sequencing.

**Model:** UiPath LLM Gateway, model `[DEFAULT]` `gemini-2.5-flash`-equivalent or platform-default fast model — confirm exact model identifier available in the org's LLM Gateway catalog at build time (`[SME REVIEW]`).

**System prompt** (ported from `SYSTEM_PROMPT` in `ai_service.py`):
- Act as a call centre analytics expert for Acme
- Back every answer with specific numbers from the injected context
- Format in markdown: headers, bold key figures, bullet points
- Apply business rules: `compliance_flag='No'` means the **agent** failed, not the customer; `call_sentiment`: 1=Positive, 0=Neutral, -1=Negative
- Know agent specialisations: Sam=Claims/Grievances, John=Policy Services, David=Escalations/Quality, Mike=New Business/Renewals, Mary=Billing/Amendments

## 3. Tools

| Tool | Input | Output | Maps to BPMN / existing |
|---|---|---|---|
| `extract_sentiment` | transcript text | `call_sentiment` (-1/0/1) | `Task_ExtractSentiment` |
| `classify_intents` | transcript text | `call_intent1/2/3` (≤3 of 9 categories) | `Task_ClassifyIntents` |
| `detect_trigger_words` | transcript text | `triggerword_flag` (Yes/No), `triggerwords` (CSV) | `Task_DetectTriggerWords` |
| `check_compliance` | transcript text | `compliance_flag` (Yes=passed, No=agent failed) | `Task_CheckCompliance` |
| `generate_followups` | transcript text | `followup_item1/2/3` (≤3, ≤100 chars each) | `Task_GenFollowups` |
| `determine_resolution_flags` | transcript text | `call_resolved_flag`, `escalation_flag`, `preverified_flag`, `repeatcall_flag` (all Yes/No) | `Task_DetermineResolution` |
| `generate_summary` | transcript text | `call_summary` (2-4 sentence plain English) | `Task_GenSummary` |
| `sentiment_analytics` | date range, agent filter | daily/monthly trend + by-agent breakdown | `Task_SentimentAnalytics` |
| `escalation_analytics` | date range, agent filter | by-agent rate, trend, root-cause (trigger words × intent) | `Task_EscalationAnalytics` |
| `compliance_analytics` | date range, agent filter | by-agent fail rate, trend | `Task_ComplianceAnalytics` |
| `resolution_analytics` | date range, agent filter | resolution/repeat-call/preverified overview + trends | `Task_ResolutionAnalytics` |
| `intent_analytics` | date range, agent filter | Pareto, trend, marketing-opportunity scoring | `Task_IntentAnalytics` |
| `friction_analytics` | date range, agent filter | composite friction score per intent, trigger-word counts/trend | `Task_FrictionAnalytics` |
| `ask` | natural-language question | markdown answer + raw data context | `Task_GenerateAIResponse` / `ai_service.ask()` |

### Tool Invocation Policy

- The 7 extraction tools are called once per incoming transcript, in parallel, by `call-center-orchestration`.
- The 6 analytics tools are called once per dashboard-refresh cycle, in parallel.
- `ask` is called synchronously, on-demand, directly from `call-center-dashboard`'s AI Insights page (not from the Flow) — it is the only tool with a human-in-the-loop-style direct caller rather than a Flow node.
- All 13 non-`ask` tools read/write Data Fabric directly (via the Data Fabric SDK/connector inside the Coded Agent) rather than receiving data as Flow variables, mirroring the existing pattern where `ai_service.py`/`analytics_service.py` query the DB themselves.

## 4. Memory / RAG

### Short-term Memory

None required — each tool call is stateless; the LLM receives only the current transcript (extraction tools) or the current aggregated context (`ask`).

### Long-term Memory

None. `[DEFAULT]` — the existing system has no agent memory; every call is independently analysed and every Q&A question is independently grounded.

### RAG Sources

`ask`'s grounding context is **not** vector-retrieval RAG — it is a structured plain-text snapshot assembled by 6 live SQL/Data Fabric aggregation queries (`_build_context()` equivalent), covering: overall KPI summary, top 8 intents, per-agent performance, monthly trend, top 10 trigger words, top 5 friction points. This must be preserved exactly as the grounding mechanism — replacing it with vector RAG would lose the guarantee that every answer cites *current* numbers.

## 5. Evaluation Criteria

### Success Metrics

- Extraction tools: field-level accuracy against a held-out labeled transcript set (`[SME REVIEW]` — no labeled eval set currently exists; the existing system has no automated extraction accuracy tracking)
- `ask`: answers must cite at least one specific figure present in the injected context (no hallucinated numbers)

### Trajectory Evaluation

Not applicable — no multi-step planning trajectory; each tool call is single-shot.

### LLM-Judge Criteria

For `ask`: an LLM-judge rubric scoring (1) groundedness in the provided context, (2) markdown formatting compliance, (3) directness/actionability — mirrors the existing `SYSTEM_PROMPT` instructions.

## 7. Error Handling & Escalation

### Agent-level errors

| Scenario | Handling |
|---|---|
| LLM Gateway timeout/rate-limit | `[DEFAULT]` retry twice with backoff; surface failure to caller (Flow or Coded App) rather than silently returning empty fields |
| Malformed/empty transcript | Return a structured error rather than guessing; the Flow's error handling (§8 of the orchestration SDD) decides whether to dead-letter |
| Data Fabric read failure (analytics tools) | Return partial results with a `data_incomplete` flag rather than failing the whole `ask`/analytics call |

### HITL Escalation

None owned by this agent — all human escalation (supervisor approval) lives in `call-center-orchestration`'s HITL nodes.

## 8. Integrated Components

### RPA Processes Called as Tools

None.

### API Workflows Called as Tools

None.

### Integration Service Connectors

None — Data Fabric and LLM Gateway access use their native SDKs/connectors, not Integration Service connectors.

**Note on write ownership:** extraction tools (the first 7) return their results to the calling Flow node rather than writing to Data Fabric themselves — `call-center-orchestration`'s `AIParallelJoin` → `StoreCallRecord` performs the single atomic write of all 7 fields together, exactly mirroring the existing rule that the join "prevents partial records ... all `call_records` fields must be populated in one atomic INSERT." This agent must NOT write `CallRecord` rows directly.

### `[SME REVIEW]` — AiUsage singleton-row pattern

The existing `ai_usage` table is a single row (`id=1`) updated via PostgreSQL `ON CONFLICT DO UPDATE`. Data Fabric does not have a native upsert-by-fixed-PK pattern identical to this; the agent's `data_fabric_client.py` must implement an explicit "read row 1, update if exists else create" pattern, with a day-aware reset of `calls_today` when `period_date != today`. Confirm this approach at build time.

## 10. Testing Strategy

### Canonical Test Case

Feed a known transcript (one of the sample `.txt` files referenced in `PROJECT_SUMMARY.md`'s `*.txt` samples) through all 7 extraction tools; assert every field is populated and within its documented domain (e.g. `call_sentiment` ∈ {-1,0,1}).

### Evaluation Dataset

`[SME REVIEW]` — build a labeled set from a sample of existing transcripts once available; until then, manual spot-checking against the existing Gemini outputs (if historical results are exported) is the fallback.

### Regression Strategy

Re-run the canonical test case after any system-prompt or model change; diff extraction outputs against the previous run to catch silent behavior drift.
