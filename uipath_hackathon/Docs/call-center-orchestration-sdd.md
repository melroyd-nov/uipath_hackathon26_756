## 1. Flow Overview

Orchestrates the full per-call analytics pipeline described in `call-center-transcript-process.bpmn`: from a transcript landing in the storage bucket through 7-way AI extraction, Data Fabric persistence, follow-up seeding, conditional supervisor approval (HITL), KPI dashboard refresh, and 6-module analytics fan-out. Grounded Q&A is a separate, user-initiated capability owned entirely by `call-center-ai-agent`/`call-center-dashboard` — see §1 Out of Scope.

### In Scope

- Trigger on new files in `cc-transcripts-incoming` (or daily batch)
- Transcript parsing handoff to the Agent (header/body split is delegated, not done in Flow)
- Parallel fork/join for the 7 AI extraction tasks and the 6 analytics modules (mirrors `GW_AIParallelFork`/`Join` and `GW_AnalyticsFork`/`Join`)
- Data Fabric writes for `CallRecord` and seeded `CallFollowup` rows
- Conditional branch on "has follow-ups" (`GW_HasFollowups`) that starts the follow-up process, fire-and-forget
- Dashboard-refresh marker (re-aggregation, not a UI render — the UI reads Data Fabric directly)

### Out of Scope

- The AI model calls themselves (delegated to `call-center-ai-agent`)
- UI rendering (owned by `call-center-dashboard`)
- Telephony recording (`Task_RecordCall` — external system, file-drop is the integration boundary)
- The follow-up approval/assignment/tracking/completion lifecycle (owned by `call-center-followup-case`, a Maestro BPMN project — see v1.1/v1.2 in Document History above)
- Grounded Q&A (`ask`) — called directly by `call-center-dashboard`'s AI Insights page, not part of this Flow (see v1.3 in Document History above)

## 3. Nodes Inventory

| Node | Type | Maps to BPMN ID | Responsibility |
|---|---|---|---|
| `TriggerNewTranscript` | Trigger | `Start_CallReceived` | Fires on new file in `cc-transcripts-incoming` |
| `IngestTranscript` | Connector (Agent tool call) | `Task_IngestTranscript` | Delegates header/body parsing to Agent; outputs `{metadata, dialogue_text, file_name}` |
| `AIParallelFork` | Parallel Gateway | `GW_AIParallelFork` | Fans out to 7 Agent tool-call nodes |
| `ExtractSentiment` … `GenSummary` (×7) | Connector (Agent tool call) | `Task_ExtractSentiment` … `Task_GenSummary` | One node per AI extraction task; each calls the matching Agent tool |
| `AIParallelJoin` | Parallel Gateway | `GW_AIParallelJoin` | Waits for all 7 before any Data Fabric write |
| `StoreCallRecord` | Connector (Data Fabric) | `Task_StoreCallRecord` | Inserts one `CallRecord` row, all 29 fields |
| `SeedFollowups` | Connector (Data Fabric) | `Task_SeedFollowups` | Inserts up to 3 `CallFollowup` rows; skips if already seeded (dedup) |
| `HasFollowupsGateway` | Exclusive Gateway | `GW_HasFollowups` | Branches on "≥1 non-null `followup_item`" |
| `StartFollowupProcess` | Connector (Maestro BPMN, fire-and-forget) | `Task_SupervisorReview` … `Task_MarkCompleted` (now owned by `call-center-followup-case`) | Starts one `call-center-followup-case` BPMN process instance per qualifying `CallRecord`; does **not** wait for it to resolve — the entire approve/assign/track/complete lifecycle (formerly `SupervisorReview`/`ApproveRejectGateway`/`AssignDetails`/`RejectFollowup`/`TrackProgress`/`ProgressGateway`/`MarkCompleted`) now lives in that project, see `call-center-followup-case-sdd.md` |
| `DashboardRefresh` | Script/no-op | `Task_UpdateDashboard` | Marker node — actual KPI computation happens on-read in the Coded App; this node exists to converge the "has follow-ups" and "no follow-ups" paths before analytics fan-out |
| `AnalyticsFork` | Parallel Gateway | `GW_AnalyticsFork` | Fans out to 6 Agent analytics tool calls |
| `SentimentAnalytics` … `FrictionAnalytics` (×6) | Connector (Agent tool call) | `Task_SentimentAnalytics` … `Task_FrictionAnalytics` | One node per analytics module |
| `AnalyticsJoin` | Parallel Gateway | `GW_AnalyticsJoin` | Waits for all 6 before pipeline completion |
| `End` | End Event | `End_Complete` | Pipeline complete |

`Task_QueryAI`/`Task_GenerateAIResponse` (BPMN IDs) are **not** modeled as Flow nodes — Q&A is called directly by `call-center-dashboard` against `call-center-ai-agent.ask()`, bypassing this Flow entirely (removed in v1.3, see Document History).

## 8. Error Handling

| Scenario | Handling |
|---|---|
| Transcript file unreadable/corrupt | Log and route to a dead-letter folder in the storage bucket; do not advance the pipeline; alert via `[DEFAULT]` Orchestrator queue-retry pattern |
| One of the 7 AI extraction calls fails/times out | `[DEFAULT]` retry twice with exponential backoff; if still failing, persist the `CallRecord` with that field NULL and flag `[SME REVIEW]` — the BPMN documentation states the join must wait for all 7 before any write, so a genuinely permanent failure needs a documented partial-record policy |
| Data Fabric write conflict (duplicate `call_id`) | Reject and alert — `call_id` must be unique per the as-built schema |
| `StartFollowupProcess` handoff fails | `[DEFAULT]` log and alert via Orchestrator queue-retry pattern; does not roll back the already-written `CallRecord`/seeded `CallFollowup` rows, and does not block the Flow's progress to dashboard refresh / analytics |
| Analytics module call fails | Non-fatal — proceed to pipeline completion with the remaining 5 modules' data; log the gap |

Supervisor Review HITL timeout/SLA handling now belongs to `call-center-followup-case` — see that SDD's §8 Error Handling.

## 10. Testing Strategy

### Canonical Test Case

Single transcript file (`Claims_Registration_FNOL.txt`-style, per the BPMN's file-naming convention) dropped into `cc-transcripts-incoming`; assert a `CallRecord` row is created with all 29 fields populated, follow-ups seeded if `followup_item1` is non-null, and the dashboard-refresh path reached.

### Happy Path Assertions

- All 7 AI extraction branches return before the join (no partial writes)
- `CallRecord.call_id` is unique and 8-digit
- Follow-ups with no items skip directly to dashboard refresh (`GW_HasFollowups` = No path)
- When follow-ups exist, `StartFollowupProcess` fires and the Flow proceeds to dashboard refresh/analytics without waiting for the process to resolve
- Pipeline reaches `End` once all 6 analytics branches return (or partially, per the non-fatal policy above)

### Error Path Scenarios

- Corrupt transcript file → dead-lettered, pipeline does not advance
- One AI branch times out → retry policy triggers; on exhaustion, partial-record policy applies (flagged `[SME REVIEW]` above)
- `StartFollowupProcess` handoff fails → logged/alerted, does not block or roll back the rest of the pipeline (see §8)

Approve/reject/assign/track/complete transition correctness is now tested in `call-center-followup-case-sdd.md` §10, not here.
