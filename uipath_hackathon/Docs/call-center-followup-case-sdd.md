## Recommended Scope

**Recommendation:** Maestro BPMN (`.bpmn`, built via `uipath-maestro-bpmn`) — per explicit instruction, this project stays solely Maestro BPMN for now rather than Case Management. The follow-up lifecycle (`pending` → `approved`/`rejected` → `in_progress` → `completed`) is modeled as a 2-lane BPMN process (Supervisor / Assigned Staff) with exclusive gateways and inline HITL tasks, consistent with how `call-center-transcript-process.bpmn` itself models human steps.
**Delivery model:** Cloud.
**Blocked by platform:** none.
**Why this is still a separate project from `call-center-orchestration`:** the lifecycle is multi-actor and indefinite-duration (waits on human turnaround), so it is started fire-and-forget by the orchestration Flow rather than run inline — keeping it as its own process lets `call-center-orchestration` finish its automated work without blocking on it. Only the *authoring product* for this lifecycle changed (Case Management → Maestro BPMN); the project boundary itself did not.

## 1. Process Overview

Manages the lifecycle of each AI-generated follow-up item from the moment `call-center-orchestration` seeds it (`source=ai_generated`, `status=pending`) through supervisor approval, assignment, progress tracking, and completion. One process instance is started **per `CallRecord`** (covering up to 3 `CallFollowup` rows for that call), not per follow-up item, since a supervisor reviews a call's follow-ups together.

### In Scope

- Pending-review queue of AI-generated follow-up items
- Supervisor approve/reject decision per item
- On approval: capture priority, assigned owner, due date
- Assigned-staff progress tracking: `in_progress` → `completed` with free-text completion notes
- Rejection path: preserve item with `status='rejected'` for audit, excluded from completion-rate analytics

### Out of Scope

- AI generation of the follow-up items themselves (owned by `call-center-ai-agent`, seeded by `call-center-orchestration`)
- KPI/analytics computation over follow-up data (computed on-read by `call-center-dashboard` directly from Data Fabric)

## 2. Lanes & Tasks

| Lane | Task | Maps to `call_followups.status` |
|---|---|---|
| Supervisor | `Task_SupervisorReview` — review AI-generated follow-up item(s) with call context, approve or reject | `pending` → `approved`/`rejected` |
| Supervisor | `Task_AssignDetails` — on approval, set priority, assigned owner, due date | `approved` → `in_progress` |
| Assigned Staff | `Task_TrackProgress` — acknowledge/start the approved item | (`in_progress`, unchanged) |
| Assigned Staff | `Task_MarkCompleted` — record free-text completion note, close item | `in_progress` → `completed` |

This is a 1:1 mapping of the existing `call_followups.status` enum (`pending`, `approved`, `rejected`, `in_progress`, `completed`) onto BPMN tasks — no new states introduced.

## 3. Gateway Logic

```
[Start: CallFollowup seeded, status=pending]
        │
        ▼
  Task_SupervisorReview
        │
        ▼
  GW_ApproveReject ──reject──► Task_RejectFollowup ──► End (terminal)
        │approve
        ▼
  Task_AssignDetails
        ▼
  Task_TrackProgress
        ▼
  GW_ProgressStatus ──not complete──► (wait / re-enter on next update)
        │complete
        ▼
  Task_MarkCompleted ──► End (terminal)
```

**Guard rule (carried over from the existing state machine):** a follow-up cannot move directly from `pending` to `in_progress` — it must pass through `approved` first. This mirrors the existing rule "pending follow-ups must be approved or rejected first," enforced by `GW_ApproveReject` gating entry into `Task_AssignDetails`/`Task_TrackProgress`.

## 4. Data Bindings

| Entity | Direction | Notes |
|---|---|---|
| `CallFollowup` (Data Fabric, shared with `call-center-orchestration` and `call-center-dashboard`) | Read/write | Process reads on start (seeded rows), writes after every task (`status`, `priority`, `assigned_to`, `due_date`, `completion_notes`) |
| `CallRecord` (Data Fabric) | Read-only | `Task_SupervisorReview` displays call context (transcript summary, agent, sentiment) alongside the follow-up items being reviewed |

No new entities are introduced — this project consumes the same `CallFollowup`/`CallRecord` entities defined in `acme-call-center-analytics-solution-sdd.md` §4.

## 5. HITL Touchpoints

| Touchpoint | BPMN task | Business intent |
|---|---|---|
| Supervisor Review | `Task_SupervisorReview` | Supervisor sees the AI-generated follow-up items for a call (with call context) and approves or rejects each |
| Assign Details | `Task_AssignDetails` | On approval, supervisor sets priority, assigned owner, and due date |
| Track Progress | `Task_TrackProgress` | Assigned staff marks an approved item in-progress (acknowledgement) |
| Mark Completed | `Task_MarkCompleted` | Assigned staff marks the item done and records a free-text completion note |

Field-level form layout is owned by `uipath-human-in-the-loop` at build time — this SDD specifies business intent only.

## 6. Triggers / Start Conditions

| Trigger | Type | Configuration |
|---|---|---|
| `StartFollowupProcess` | Started by `call-center-orchestration`, fire-and-forget | `call-center-orchestration`'s `HasFollowupsGateway` (Yes branch) starts one `call-center-followup-case` process instance per `CallRecord` with ≥1 non-null `followup_item`, then **does not wait** — the Flow proceeds immediately to its analytics fan-out (see `call-center-orchestration-sdd.md` §2) |

This is the key behavioral property carried over from the prior design: the Flow does not block on human turnaround time. The process runs independently; `call-center-dashboard` reads live `CallFollowup` status directly from Data Fabric regardless of where the process instance currently sits.

## 8. Error Handling

| Scenario | Handling |
|---|---|
| Process instance fails to start (Flow → BPMN handoff error) | `[DEFAULT]` Flow logs and alerts via Orchestrator queue-retry pattern; the `CallRecord`/seeded `CallFollowup` rows are not rolled back — the process is retried independently of the already-completed AI pipeline |
| `Task_SupervisorReview` times out / unassigned | `[DEFAULT]` escalate to a configurable fallback approver after 48 hours — no SLA was specified in the BPMN/`PROJECT_SUMMARY.md`, flagged as `[SME REVIEW]` |
| Invalid transition attempted (e.g. `pending` → `in_progress` directly) | Rejected by `GW_ApproveReject` (§3) |

## 10. Testing Strategy

### Canonical Test Case

Start a process instance for a `CallRecord` with 2 non-null `followup_item`s; approve one, reject the other; assign priority/owner/due-date to the approved item; progress it to `in_progress` then `completed` with a completion note; assert `CallFollowup` rows reflect the correct terminal states and the rejected item is excluded from completion-rate calculations.

### Happy Path Assertions

- Process starts automatically when `call-center-orchestration` seeds ≥1 follow-up, without the Flow waiting for completion
- `pending` → `in_progress` direct transition is rejected (`GW_ApproveReject` guard enforced)
- Rejected items are preserved (not deleted) and flagged for audit exclusion

### Error Path Scenarios

- Process-start handoff failure does not roll back the already-persisted `CallRecord`/seeded `CallFollowup` rows
- Unassigned `Task_SupervisorReview` past 48h triggers fallback escalation (flagged `[SME REVIEW]`)
